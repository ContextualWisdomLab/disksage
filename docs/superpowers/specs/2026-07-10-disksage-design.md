# DiskSage — Design Spec

- **Date**: 2026-07-10
- **Brand**: DiskSage (disksage.com / disksage.app — RDAP 확인 결과 미등록, 조속히 등록 필요)
- **Repo**: `ContextualWisdomLab/disksage`
- **Status**: Approved by owner (brainstorming session 2026-07-10)

## 1. 개요와 목표

Windows / Linux / macOS 데스크톱에서 디스크 공간을 파악하고 확보하는 앱.
개인용 도구로 시작해 공개 배포용 제품으로 진화한다 (ContextualWisdomLab 브랜딩).

핵심 기능:

1. **대용량 파일/폴더 탐색** — 병렬 스캔 + 트리맵/목록 시각화
2. **알려진 캐시/임시파일 정리** — OS·브라우저·패키지 매니저 캐시 원클릭 정리
3. **개발 아티팩트 정리** — 오래된 `node_modules`, `target/`, `venv` 등
4. **중복 파일 탐지** — 크기 → 부분 해시 → 전체 해시 3단계
5. **온톨로지 기반 파일 재배치(정리정돈)** — OWL 분류체계에 따라 파일 분류 이동
6. **디스크 인벤토리 뷰** — "내 디스크에 뭐가 있는지"를 온톨로지 클래스별로 집계
7. **온디바이스 LLM 판별** — 내장 소형 모델이 "지워도 되는 파일인가"를 자문(advisory)

## 2. 비목표 (v1)

- OWL 추론(reasoning) 엔진 — 포맷은 OWL, v1은 클래스 계층 순회만. 추론기는 향후 확장
- 코드 서명·자동 업데이트 — 인증서 필요, 1.0 이전 과제로 문서화만
- 영구 삭제 — v1에는 존재하지 않음. 모든 삭제는 OS 휴지통 경유
- 백그라운드/예약 정리 — 사용자가 실행할 때만 동작

## 3. 기술 스택

| 계층 | 선택 | 이유 |
|------|------|------|
| 셸 | Tauri 2 | 3 OS 지원, ~10-15MB 바이너리, 코드서명/업데이터 내장 |
| 백엔드 | Rust | 병렬 스캔 성능, llama.cpp 네이티브 바인딩 |
| 프론트엔드 | Svelte 5 + TypeScript + Vite | 경량, Tauri 기본 템플릿 |
| LLM | `llama-cpp-2` + GGUF 소형 모델 (~1B, Q4) | 완전 오프라인, 첫 사용 시 다운로드 |
| 온톨로지 | OWL (Turtle), `sophia` 크레이트로 파싱 | W3C 표준, Protégé 호환 |
| 휴지통 | `trash` 크레이트 | OS별 휴지통 추상화 |
| 해시 | `blake3` | 중복 탐지 속도 |

## 4. 아키텍처

Tauri 2 단일 앱. Rust 백엔드가 무거운 작업(스캔·해시·LLM·파일 조작) 전부를 담당,
프론트엔드는 시각화와 사용자 승인만 담당. 통신은 Tauri command(요청/응답) +
event(진행률 스트리밍).

### Rust 코어 모듈

각 모듈은 독립적으로 테스트 가능한 단위이며, 프론트엔드는 모듈 내부를 모른다.

| 모듈 | 책임 | 의존 |
|------|------|------|
| `scanner` | `jwalk` 병렬 순회로 용량 트리 구축, 진행률 이벤트 스트리밍. 심링크/정션 미추적 | — |
| `rules` | OS별 알려진 캐시/임시 경로 정적 테이블 + 개발 아티팩트 탐지(프로젝트 마커 인접 + mtime 기준) | scanner |
| `dupes` | 크기 그룹핑 → 부분 해시 → blake3 전체 해시 | scanner |
| `ontology` | OWL Turtle 파싱 → 클래스 계층 트리 제공 | — |
| `organize` | 파일 → 온톨로지 클래스 매핑(저비용 신호 + LLM), 목적지 결정 | ontology, llm |
| `inventory` | 스캔 결과를 온톨로지 클래스별 집계, 미분류 버킷 관리 | scanner, organize |
| `llm` | llama.cpp 추론: 삭제 판정 + 분류 선택 + 미분류 뭉치 요약. 판정 캐시 | — |
| `safety` | 휴지통 경유 삭제, 원자적/검증 이동, 보호 경로 거부, 작업 저널·되돌리기 | — |

### 프론트엔드 화면

1. **트리맵/대용량 뷰** — 스캔 결과 시각화, 드릴다운
2. **정리 카테고리 뷰** — 캐시 / 개발 아티팩트 / 중복 각각의 후보 목록
3. **인벤토리 뷰** — 온톨로지 클래스별 용량 집계, 미분류 버킷 강조, LLM 요약
4. **검토 화면** — 선택 항목 + LLM 판정 배지 확인 후 일괄 실행
5. **정리정돈 뷰** — 파일 → 클래스 → 목적지 매핑 미리보기 후 승인

## 5. 온톨로지 설계

- **포맷**: OWL, Turtle 직렬화. `owl:Class` + `rdfs:subClassOf` 계층
- **어노테이션**: `rdfs:label`(ko/en), `dm:targetFolder`(이동 목적지 템플릿, 예: `~/Documents/영수증/{year}/`)
- **네임스페이스**: `dm:` = `https://disksage.app/ontology#`
- 앱이 기본 온톨로지 제공(문서>계약서/영수증/보고서, 미디어>사진/영상, 설치파일, 코드, 데이터셋 등)
- 사용자 설정 디렉토리의 `.ttl` 파일을 직접 편집하거나 교체 가능. 표준 OWL이므로 Protégé로도 편집 가능
- `dm:targetFolder`가 없는 클래스는 가장 가까운 상위 클래스의 목적지를 상속

### 분류 파이프라인

```
파일 → ① 저비용 신호(확장자·MIME·파일명 패턴)로 후보 클래스 압축
     → ② LLM이 후보 목록 중에서만 클래스 ID 선택 (자유 생성 금지)
     → ③ 클래스의 dm:targetFolder로 목적지 결정
     → ④ 미리보기 승인 후 이동
```

### 인벤토리 뷰

"파일이 뭐가 있는지 모르겠다"에 답하는 화면. 온톨로지 클래스별 용량 집계.
**미분류(Unknown) 버킷은 일급 시민** — 분류 안 되는 용량이 숨겨지지 않고 드러나며,
LLM에 샘플 배치를 보내 "이 뭉치는 무엇인가" 요약을 받을 수 있다.

## 6. LLM 설계

- `llama-cpp-2` 크레이트로 llama.cpp 내장 (별도 프로세스·설치 없음)

### GPU 가속 매트릭스

| 하드웨어 | 백엔드 | 비고 |
|----------|--------|------|
| Apple Silicon (macOS) | llama.cpp **Metal** | 기본 활성. GGUF 추론에서 MLX와 동급의 Apple GPU 가속 |
| Nvidia (Windows/Linux) | llama.cpp **CUDA** | CUDA 런타임 감지 시 사용 |
| AMD/Intel GPU | llama.cpp **Vulkan** | CUDA 미감지 시 폴백 |
| GPU 없음 | CPU | 항상 동작하는 최종 폴백 |

- 런타임에 사용 가능한 백엔드를 자동 감지, 실패 시 단계적으로 CPU까지 폴백 — 사용자 설정 불필요
- 별도 MLX 프레임워크 통합은 비채택: 제2의 추론 엔진과 별도 모델 포맷(MLX ≠ GGUF)이 필요해져 복잡도 대비 이득이 없음. Metal 백엔드가 동일 목적(Apple GPU 가속)을 달성. MLX 네이티브가 꼭 필요해지면 v2에서 재검토
- 모델: 기본 후보 Qwen3-1.7B-Instruct GGUF Q4 (약 1.1GB), 경량 대안 Llama-3.2-1B (약 0.8GB). 구현 계획 단계에서 분류 정확도 벤치 후 하나로 확정. 첫 사용 시 앱 데이터 디렉토리로 다운로드, SHA-256 검증
- 입력: 파일 메타데이터(경로, 이름, 크기, mtime, 부모 컨텍스트). **파일 내용은 보내지 않음** (첫 바이트 기반 타입 감지는 로컬 시그니처로)
- 출력: 강제 JSON `{"verdict": "safe"|"caution"|"keep", "reason": "..."}` — temperature 0, 판정 결과 로컬 캐시
- **자문 역할만**: 판정은 UI 배지일 뿐, 자동 삭제 트리거가 될 수 없다
- 모델이 없거나 다운로드 실패 시 앱은 규칙 기반으로 완전히 동작 (배지만 "미판정")

## 7. 안전 모델

파괴적 작업의 불변식:

1. 삭제 = 항상 OS 휴지통 경유. 영구 삭제 코드 경로가 v1에 존재하지 않는다
2. 이동 = 같은 볼륨이면 원자적 rename, 다른 볼륨이면 복사 → 크기/해시 검증 → 원본 휴지통
3. 시스템 경로·앱 설치 경로는 하드코딩 거부 목록 — 어떤 요청도 통과 불가
4. 모든 작업은 실행 **전에** 저널(JSON)에 기록 → 크래시 시 진행 지점 복구 가능
5. "마지막 작업 되돌리기" — 이동은 저널로 역이동, 삭제는 휴지통 복원 안내
6. 모든 실행은 검토 화면에서 사용자의 명시적 승인 후에만

## 8. 에러 처리

- 권한 없음/사용 중 파일 → 건너뛰고 집계, UI에 "스킵 N건" 표시. 스캔은 중단되지 않음
- 휴지통 이동 실패 → 해당 항목만 중단·보고. 영구 삭제 폴백 없음
- Windows 긴 경로 → `\\?\` 프리픽스 처리
- LLM 실패 → 기능 저하(배지 미표시)일 뿐 앱 동작에 영향 없음

## 9. 테스트

- Rust 유닛 테스트: `scanner`/`rules`/`dupes`/`ontology`/`organize` — tempdir 픽스처
- 통합 테스트 1: 휴지통 왕복(삭제→복원)
- 통합 테스트 2: 크로스 볼륨 이동 검증 로직 (동일 볼륨 시뮬레이션)
- `llm` 모듈: 프롬프트 생성·JSON 파싱 스텁 테스트. 실 모델 테스트는 `--ignored` 뒤로
- CI: GitHub Actions 매트릭스 (windows-latest / ubuntu-latest / macos-latest) 빌드 + 테스트
